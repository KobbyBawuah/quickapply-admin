Build a modern local-only admin web application for QuickApply Pro email automation.

PROJECT GOAL
Create a private internal dashboard used by the client locally. The dashboard connects to the client’s MongoDB database where QuickApply Pro users are stored. It should detect inactive users, generate/send marketing emails, schedule recurring campaigns, and keep a complete history of which user received which email.

This app is for internal/local use only. It is not a public SaaS app.

BRAND CONTEXT
QuickApply Pro is an AI-powered job application platform. It helps users:

- Tailor resumes in under 60 seconds
- Use a Chrome extension to autofill job application forms
- Generate tailored cover letters
- Grade resumes against job descriptions
- Prepare for interviews
- Detect fake/scam job postings
- Support international job seekers
- Apply faster while keeping quality high
- Access an affordable alternative compared to competitors

Brand tone:
- Sharp, confident, clean, direct
- Not desperate, not spammy, not overhyped
- Outcome-first copy
- Professional but engaging
- Do not fabricate claims
- Do not use generic words like “game-changer”, “dream job”, “supercharge”, “cutting-edge”

5. Inactive User Campaign Page

Purpose:
Every Monday, collect users who have logged in before but have been inactive for more than 7 days. Send them a persuasive reactivation email encouraging them to purchase or renew a subscription.

Logic:
Query users where:
- email exists
- lastLoginAt exists
- lastLoginAt older than 7 days
- subscriptionStatus is free, trial, expired, or inactive
- doNotContact is not true
- user has not received the same campaign in the last 7 days

Features:
- Show matching users before sending
- Allow manual “Run Campaign Now”
- Allow scheduled automatic run every Monday at 9:00 AM
- Store each campaign run in campaign_runs
- Store every email sent in email_logs
- Show sent, failed, and skipped users

IMPORTANT SAFETY

Before bulk sending:
- Show confirmation modal
- Show number of recipients
- Require admin to type SEND to confirm
- Do not send to users without email
- Do not send to doNotContact users
- Do not send duplicate campaign to same user within cooldown period
- Do not send more than MAX_EMAILS_PER_RUN in one campaign run
- Apply DELAY_BETWEEN_EMAILS_MS between each email
- Log skipped users and reason
